# /debug $ARGUMENTS

## Step 1 — 재현 확인
- 기대 동작과 실제 동작을 한 문장씩 서술
- 버그가 발생하는 진입점 특정 (어느 컴포넌트, 라우트, 사용자 액션)

## Step 2 — 추적
- 진입점에서 실패 지점까지 데이터 흐름 추적
- 관련 파일 전체 읽기 (가설 세우기 전에 읽기 완료)
- 가능한 원인 모두 나열

## Step 3 — 격리
- 잘못된 동작이 시작되는 최소 단위(함수, 훅, 컴포넌트) 특정
- 가설 서술: "X에서 Y 때문에 버그가 발생한다고 판단"

## Step 4 — 수정
- 근본 원인만 최소한으로 수정 — 주변 코드 리팩토링 금지
- 타입 에러가 런타임 에러를 숨기고 있다면 런타임 에러 먼저 수정

## Step 5 — 검증
- 수동 확인 방법 또는 실행할 테스트 명시
- 수정이 인접 동작(유사한 컴포넌트, 형제 라우트)을 깨지 않는지 확인
- 이 영역을 커버하는 테스트가 있다면 실행하여 통과 확인

## Step 6 — 평가자용 설명
- 버그가 무엇이었는지, 왜 발생했는지, 수정이 어떻게 동작하는지 1단락으로 서술
