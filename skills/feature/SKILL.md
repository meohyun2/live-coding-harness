# /feature $ARGUMENTS

## Step 1 — 범위 파악
- 기능 요건을 자신의 말로 재서술하여 이해 확인
- 수정될 기존 파일과 새로 생성할 파일 목록 작성
- 새 의존성이 필요한지 확인 (필요 시 설치 전에 확인)

## Step 2 — 기존 패턴 탐색
- 구현할 기능과 가장 유사한 기존 기능 탐색
- 해당 컴포넌트 + 훅 + 타입 + 앱 연결 방식을 함께 읽기
- 네이밍 컨벤션, 파일 위치, export 방식 메모

## Step 3 — 구현 계획 서술 (코드 작성 전)
1. 정의할 새 타입/인터페이스
2. 생성할 커스텀 훅 (필요 시)
3. 신규/수정 컴포넌트
4. 라우팅 변경 (필요 시)
5. API 연동 포인트 (필요 시)

## Step 4 — 구현
- 순서: 타입 → 훅 → 컴포넌트 → 연결
- 각 파일 완료 후 완료된 것과 남은 것 서술
- 기존 패턴 정확히 따르기 (네이밍, export, 파일 위치, 스타일링)

## Step 5 — 통합 확인
- 기능이 UI에서 접근 가능한지 확인 (라우트, 링크, 버튼)
- `npx tsc --noEmit`으로 TypeScript 컴파일 확인
- 기능의 수동 스모크 테스트 방법 서술
